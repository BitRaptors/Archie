# ui/
> Unstyled, composable UI component library using CVA variants and Tailwind; extend via className prop.

## Patterns

- All components use React.forwardRef to expose DOM refs for parent control and testing
- CVA (class-variance-authority) defines style variants; cn() merges base + variant + custom classes
- Compound components (Card + CardHeader + CardTitle + CardContent + CardFooter) share semantic structure
- Components extend native HTML attributes (HTMLButtonElement, HTMLDivElement, etc.) for full DOM compatibility
- Slot pattern (button.tsx) via asChild prop allows component polymorphism without wrapper divs
- displayName set on every forRef component for React DevTools clarity in debugging

## Add new variant to existing component (e.g., button size or color)

1. Add variant key+value to cva() variants object (e.g., buttonVariants)
2. Update defaultVariants if needed
3. Test: component accepts new variant prop with TypeScript support auto-generated by CVA

## Usage Examples

### Compound component + forRef pattern (from card.tsx)
```typescript
const Card = React.forwardRef<HTMLDivElement, React.HTMLAttributes<HTMLDivElement>>
(({ className, ...props }, ref) => (
  <div ref={ref} className={cn("rounded-lg border bg-card shadow-sm", className)} {...props} />
))
Card.displayName = "Card"
```

## Don't

- Don't hardcode Tailwind classes in components — always wrap with cn() to allow className override
- Don't forget displayName on forRef components — breaks DevTools debugging and error messages
- Don't add new variants without updating both CVA object AND interface VariantProps — TypeScript won't catch mismatches

## Testing

- Render component with className prop and verify custom classes appear in DOM (cn() merge works)
- Pass ref via React.createRef; confirm ref.current points to correct HTML element (forRef wired correctly)

## Debugging

- displayName missing? Component shows as 'ForwardRef' in React DevTools — copy pattern from badge.tsx
- Variant not applying? Check CVA defaults and verify component received correct variant prop value

## Why It's Built This Way

- forRef everywhere: allows focus management, imperative DOM APIs, and parent style overrides without wrapper
- CVA over inline if/switch: type-safe variants, single source of truth for styles, easier to audit design tokens
